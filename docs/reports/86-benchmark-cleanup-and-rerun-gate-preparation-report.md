# Task 86 — Benchmark Cleanup, Claim-Safety Fixes, and Rerun Gate Preparation Report

## 1. Objective
Clean up the benchmark state following Tasks 83–85, finalize claim-safety boundaries for quality proxy limitations and compression latency constraints, and establish a strict, controlled rerun sequence (manifest and checklist) for the upcoming validation benchmarks (Tasks 87 and 88).

## 2. Why Task 86 Was Needed
The previous tasks (84 and 85) clarified and bounded our claims: QMSum cannot support semantic correctness claims due to its abstractive references, and the current large LLMLingua-2 compressor could not be forced onto GPU in the tested 8GB setup without crashing. While these were necessary diagnostics, they introduced feasibility research paths that distracted from a clean release. Task 86 formalizes these limits, ensures the Roadmap does not prematurely jump to drafting the final report, and implements a controlled `n=10` (Task 87) and `n=30` (Task 88) sequence to ensure the rerun artifacts are complete, comparable within the selected setting, and pass basic execution-quality checks before moving on.

## 3. Task 84 & Task 85 Wording Cleanup
A review of the artifacts generated by Tasks 84 and 85 confirmed that the existing wording strictly adhered to claim-safe boundaries:
- **Task 84**: Clearly stated that QMSum remains diagnostic-only, makes no final quality claims, and notes that strict zero-shot extractive prompting correctly yields low overlap against abstractive references.
- **Task 85**: Clearly stated that forcing the current large compressor onto GPU is unsafe in the tested 8GB setup, and offline/cached compression is positioned correctly as an upper-bound deployment mode hypothesis.

No retroactive patches to these reports were necessary.

## 4. Roadmap Correction
The Roadmap was previously pointing prematurely to Task 86 (or Task 85) as the "Final Report Drafting" phase. It has been updated to explicitly point to the newly established **Task 87 — Controlled n=10 benchmark rerun gate**. 

## 5. Rerun Manifest
A formal execution plan has been locked in `results/task86_rerun_gate_manifest.json`.
- **Target Datasets**: `gsm8k_short` and `qmsum_meeting_qa_long`
- **Conditions**: Baseline-AR, DFlash-R1, LLMLingua-AR-R2, CC-DFlash-R2
- **Parameters**: `seed=42`, `max_new_tokens=512`, `keep_rate_percent=50`, `prompt_source="dataset"`, `prompt_policy="strict zero-shot"`

> [!NOTE]
> Task87/Task88 use `max_new_tokens=512` as the controlled rerun setting. Task83 remains the repaired reference under its original setting and should not be treated as an identical-latency comparison against the 512-token rerun.
> Task87/Task88 use `max_new_tokens=512` as the controlled rerun setting. Task83 remains the repaired reference under its original setting and should not be treated as an identical-latency comparison against the 512-token rerun.

## 6. Validation Checklist
A strict set of gate criteria has been codified in `results/task86_rerun_validation_checklist.json` to prevent silent benchmark failures, including:
- **Row count**: Must strictly equal 10 (for Task 87) or 30 (for Task 88).
- **Quality limits**: Maximum repetitions (<=2), empty outputs (0), hit cap limits (<=3).
- **Diagnostics Logging**: Ensures latency (`avg_e2e_latency`, `avg_generation_latency`, `avg_T_compress`), throughput, and VRAM are properly captured.

## 7. Gate Rule for Task 87 and Task 88
- **Task 87 (n=10)** is authorized to run. It will validate that the benchmark execution scripts produce the correct output shapes without stalling or crashing.
- **Task 88 (n=30)** is strictly gated. It may **only** proceed if Task 87 passes all criteria defined in the Task 86 validation checklist. The companion script `scripts/analyze_task86_rerun_gate.py` can be used to automatically verify the success of these gates.
