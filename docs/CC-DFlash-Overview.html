1. Abstract & Executive Summary

CC-DFlash proposes an independent input-compression layer placed before the DFlash speculative decoding pipeline. The compressor receives Cgốc (N tokens) and produces Cnén (N/R tokens) as natural text — preserving vocabulary compatibility with Qwen3 without introducing meta-tokens or modifying DFlash internals.

Core Claim — Read Carefully
Lossy input compression + lossless speculative decoding on compressed context ≠ lossless full pipeline. CC-DFlash does not claim end-to-end losslessness.
What CC-DFlash proposes: Reduce prefill cost for long-context workloads by compressing the input context before DFlash. The speculative decoding algorithm is unchanged. Quality depends on how much answer-relevant evidence survives compression. End-to-end speedup depends on whether prefill savings exceed compression overhead.
What has not been proven (after Task 33): (1) Confirmed 8 GB deployment — VRAM not yet measured under load. (2) Final speedup — pilot data is n=6–20, preliminary. (3) Compression proven useful end-to-end — CPU T_compress overhead may dominate in short/medium context. (4) R3 quality safe — containment data suggests potential harm at aggressive compression.

2. Problem Statement

Long-context inference with large language models is dominated by two costs: prefill (processing all input tokens through the full network) and decoding (generating output tokens autoregressively or speculatively). Speculative decoding approaches like DFlash address the decoding stage efficiently but do not reduce the cost of processing a long input context during prefill.

Prefill Cost
O(N²) attention

Scales quadratically with token count N. Dominant for long-context workloads.

DFlash Contribution
Accelerated decoding

Block-diffusion speculative decoding generates K=16 tokens per step. Does not address prefill cost.

CC-DFlash Explores
Prefill cost reduction

Compress input to N/R tokens before prefill. Hypothesis: O((N/R)²) prefill savings may exceed T_compress overhead.

The core research question is whether extractive input-side compression can provide meaningful prefill reduction without destroying enough answer-relevant signal to harm output quality — and whether the compression CPU overhead can be offset by GPU prefill savings.

3. Research Hypothesis

H1 — Prefill Cost Reduction

Prefill cost scales approximately as O(N²). Compressing context from N to N/R tokens reduces prefill from T_prefill(N) to approximately T_prefill(N)/R². For large enough N, this saving exceeds T_compress.

H2 — Acceptance Length Preservation

DFlash's speculative decoding acceptance length τ on compressed context C_nén remains high enough to preserve useful decoding throughput. Compression may shift the hidden-state distribution, but the hypothesis is that this shift is tolerable for moderate R values.

H3 — Quality Preservation under Moderate R

At R≈2 (50% keep rate), LLMLingua-2 extractive compression preserves enough answer-relevant token sequences that output quality is not significantly harmed for factual reasoning tasks. At R≈3, quality risk increases and must be evaluated empirically.

H4 — CPU Overhead Amortization

In multi-turn or RAG scenarios where the same compressed context is reused across Q queries, T_compress is amortized by a factor of Q, reducing effective overhead toward zero.

Ttotal = Tcompress + Tprefill(N/R) + Tdraft + Tverify + Tdecode
(1)
Breakeven: Tprefill(N) · (1 − 1/R²) > Tcompress
(2)
Status: Breakeven has not been confirmed. T_prefill is now captured as a CUDA-synchronized smoke measurement in Task 39 artifacts. T_compress from CPU is captured per run, and Task 40 derives a preliminary breakeven-readiness summary. Compressed-condition T_prefill is still not measured in the existing compressed artifacts, so this remains a hypothesis pending larger controlled measurements.

4. Core Architecture

CC-DFlash inserts an extractive compression step before DFlash. The compressor runs on CPU and produces natural-language output that shares the same vocabulary and tokenizer as the Qwen3 target model. No architecture changes are made to DFlash itself.

Why there is no vocabulary mismatch: Cnén is natural text produced by extractive token selection from the same Qwen3 vocabulary. The target model receives it as a normal prompt. No meta-tokens, no latent vectors, no internal DFlash modifications. This is the key reason the CC-DFlash approach avoids vocabulary misalignment.

Thinking mode must remain OFF: Draft model Qwen3-4B-DFlash-b16 is designed for non-thinking generation. If thinking mode is active on the target, acceptance length τ drops to near zero, eliminating all DFlash benefit.

6. Compression Strategy

6.1 Principles

Context is compressible; question is protected. The segmenter separates the long context block (which can be compressed) from the question (which must never pass through the compressor). This separation prevents the compressor from destroying critical numerical or logical tokens in GSM8K-style math questions.

Output is always natural text. LLMLingua-2 performs extractive compression — it selects and removes tokens from the original text rather than rewriting. The output is always a subset of the original tokens in the same vocabulary as Qwen3.

No manual chunking. LLMLingua-2's underlying BERT model handles window management internally. Manual chunking would break global coherence. LongLLMLingua is reserved for Phase 2 at 4000+ token contexts.

6.2 Answer-Aware Compression

The question= parameter in LLMLingua-2's API biases token importance scores toward content relevant to answering the question. This is the correct strategy for preserving math-relevant evidence in GSM8K contexts. The compressor does not guarantee preservation of specific numbers — protection of the actual question is the primary safeguard.

6.3 LLMLingua-AR Baseline — Why It Is Required

Attribution separability: The LLMLingua-AR condition (LLMLingua-2 compression + autoregressive decoding, without DFlash) is mandatory. If CC-LLM-R3 outperforms DFlash-R1, but LLMLingua-AR-R3 achieves similar throughput, then the speedup comes from compression, not from DFlash. Without this baseline, any speedup claim about CC-DFlash is incorrectly attributed.

7. Benchmark Philosophy

The benchmark must be designed to answer three specific questions: (1) Does CC-DFlash improve end-to-end throughput over DFlash-R1? (2) Does speedup come from compression or from DFlash? (LLMLingua-AR attribution) (3) What is the quality cost at each compression ratio?

Answer Quality Scorer (Task 32): Deterministic containment scorer with 4 categories: EXACT (ground truth appears verbatim in output), NORMALIZED (match after lowercase/whitespace normalization), NO_CONTAINMENT (not found), and NOT_EVALUABLE (empty output or ground truth). 10.1 Carry-Forward Decision

Condition Decision Rationale
DFlash-R1 KEEP_BASELINE Primary comparison reference — always needed
LLMLingua-AR-R2 KEEP_LOW_VRAM_BASELINE Attribution baseline — separates compression vs DFlash contribution

CC-LLM-R3 WATCHLIST Possible containment harm — needs targeted manual review (Task 35)
LLMLingua-AR-R3 DEPRIORITIZE Not needed until R2 baseline is solidified

10.2 Conservative Interpretation

DFlash-R1 remains the strongest current condition in preliminary fixture data. LLMLingua-AR-R2 is useful as a low-VRAM attribution baseline. CC-LLM paths have not been demonstrated to be e2e beneficial under CPU compression overhead — this is the central open question. R3 paths may harm answer quality — manual review and larger n are required. No final speedup, correctness, or deployment claim is supported by current evidence.
