## Problem Statement

The current low-tier path uses:

```text
drafter -> text bridge -> verifier
```

The drafter currently emits text chunks from a different model family than the verifier. The bridge can determine whether a draft text chunk is structurally safe for the current diagnostic policy, but bridge-valid text is not the same as verifier-accepted target tokens.

The project therefore needs a staged correctness specification before any strict acceptance runtime is implemented.

## Tokenizer Mismatch

The drafter and verifier do not necessarily share a tokenizer.

Current block sizing:

```text
draft_block_size = 8
```

means:

```text
Qwen-side draft max tokens = 8
```

It does not mean:

```text
8 Gemma target tokens
```

It must not be interpreted as:

```text
accepted target token count
accepted Gemma token count
correctness evidence
speed evidence
```

This is why Phase 3.10 starts with a text-level candidate and a text-prefix verification unit.

## Candidate Unit

The initial candidate unit is:

```text
draft_text_chunk
```

It is generated by:

```text
drafter
```

It is bounded by:

```text
configured drafter-side draft_block_size
configured maximum character/length limits
text bridge validation
```

`draft_text_chunk` is not a target-token block.

`draft_text_chunk` is not a Gemma-token block.

`draft_text_chunk` must not be counted as accepted target tokens.

Future candidate metadata should include:

```text
candidate_text
candidate_source_role
candidate_context_hash
candidate_cycle_index
candidate_generation_settings
candidate_normalization_status
candidate_bridge_status
```

## Verification Unit

The Phase 3.10 verification unit is:

```text
normalized text prefix
```

The verifier question is:

```text
Is the normalized candidate text a prefix of the normalized verifier continuation text?
```

This is a text-level specification.

It is not token-level acceptance.

The future Phase 3.11 verification unit should be:

```text
Gemma target-token sequence
```

That future design must decide how to tokenize candidate text with the verifier tokenizer and compare it against deterministic verifier decisions.

## Deterministic Settings

Correctness/equivalence claims are meaningful only under defined deterministic settings.

Required future validation settings:

```text
temperature = 0.0
fixed seed where supported
greedy decoding
stable prompt mode
same effective prompt
same context update rule
same stop handling
same continuation bound where applicable
same comparison normalization rule
model identity recorded
model file recorded
runtime version recorded where available
stdout/stderr separated for reproducibility logs
```

The semantic comparison must not depend on terminal formatting or runtime log ordering.

## Text Normalization Boundary

Correctness comparison normalization is separate from display cleanup.

The first comparison profile should be strict:

```text
normalize CRLF/CR to LF
avoid semantic rewriting
avoid punctuation edits
avoid Unicode rewriting unless explicitly specified
avoid whitespace rewriting beyond the comparison profile
```

Response display cleanup may strip or format text for CLI display, but that cleanup must not silently become correctness comparison logic.

Do not rewrite model output to force matches.

## Acceptance Semantics

Initial text-level acceptance is strict full-candidate prefix acceptance.

A candidate is accepted at text-prefix level only when:

```text
candidate passes text bridge validation
verifier continuation is generated under deterministic settings
normalized candidate text is a prefix of normalized verifier continuation text
accepted span is recorded as text span / character span
```

The accepted span is not a target-token count.

The first prototype should use full candidate prefix acceptance before adding partial text-prefix behavior.

## Rejection Semantics

A candidate is rejected when:

```text
bridge validation fails
candidate is empty after normalization
candidate is not a prefix under the selected comparison profile
deterministic verifier continuation cannot be produced
comparison is ambiguous
candidate violates length/context constraints
```

Rejection metadata should include:

```text
rejection_reason
rejection_cycle_index
rejection_stage
fallback_triggered
context_update_source
```

Bridge rejection is not target-token rejection.

## Fallback Semantics

Fallback is a recovery mechanism.

When a candidate cannot be accepted under the active correctness profile:

```text
do not append the candidate as accepted output
generate or use verifier continuation fallback
append verifier-produced fallback text according to policy
update context only with fallback text
record fallback reason
```

Fallback count is not correctness evidence.

Fallback count is not performance evidence.

Fallback count is not a quality score.

## Partial Match Semantics

Phase 3.10 recommends strict full-candidate prefix acceptance first.

Partial text matching can be designed later as:

```text
longest common prefix in normalized text space
```

If partial text matching is added, it must be reported as a text span or character span, not as target tokens.

Token-position partial matching belongs to the future token-level verifier design.

## Context Update Semantics

Strict correctness behavior requires:

```text
next context = previous context + accepted candidate text
```

or:

```text
next context = previous context + verifier fallback text
```

Do not update context with rejected candidate text.

Do not update context with unverified drafter text.

Current v0.2 behavior is engineering behavior. Phase 3.12 should implement strict context update behavior after Phase 3.11 defines token-level verification design.

## Baseline Equivalence Target

The future low-tier correctness validation target is:

```text
Gemma E2B greedy baseline
```

Meaning:

```text
strict low-tier output should match Gemma E2B alone under the same deterministic prompt/context/settings rules
```

This target belongs to Phase 3.13 validation.

Gemma E4B is not the low-tier equivalence target for Phase 3.10.

Gemma E4B remains future target/high-tier work.

No equivalence claim is made in this phase.

## Metrics Naming Rules

Current structural metrics remain:

```text

```

These are not acceptance metrics.

Do not rename:

```text

bridge_valid_block_count -> acceptance_count
cycle_fallback_count -> success_count
cycle_fallback_count -> quality_metric
```

Future correctness metrics may use acceptance names only after semantics exist and validation code implements them.

Do not add acceptance-rate metrics in Phase 3.10.
