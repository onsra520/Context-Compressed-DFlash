Tóm tắt Điều hành

CC-DFlash đề xuất một lớp nén đầu vào độc lập đặt trước pipeline DFlash — không sửa bên trong DFlash, không thay đổi không gian từ vựng, không sinh meta-token. Compressor nhận Cgốc (N tokens), sinh Cnén (N/R tokens) ở dạng text tự nhiên, rồi đưa Cnén vào cặp Target/Draft như prompt thông thường.

Claim chính xác của hệ thống: lossy input compression + lossless speculative decoding trên compressed context. Không phải lossless end-to-end.

Verdict DRR (báo cáo kiểm toán chính thức): Khả thi như một prototype nghiên cứu — đặc biệt nếu đi theo đường ít rủi ro nhất là LLMLingua-2 + Qwen3-4B DFlash non-thinking + backend Transformers. Chưa khả thi như một claim "đã chốt kiến trúc deploy" vì nhiều giả định triển khai chưa được xác minh thực nghiệm.
Lỗi lớn nhất của v3.0 (đã sửa trong v4.0): Con số VRAM 2.5 GB cho Target và 1.5 GB cho Draft ở FP16/BF16 là sai — Qwen3-4B ở FP16 chiếm ~8 GB chỉ riêng trọng số. v4.0 sửa lại với số liệu đúng và chiến lược quantization rõ ràng.

3. Synthetic Probe — Blocker Trước Mọi Thứ Khác

Đây là bước duy nhất cần chạy và pass trước khi commit vào bất kỳ benchmark nào. Nếu Probe fail, toàn bộ pipeline CC-DFlash không hoạt động và mọi số liệu tiếp theo vô nghĩa.

CRITICAL PATH — Chạy cái này trước, không có ngoại lệ

Câu hỏi cần trả lời: Khi Target Qwen3-4B được load ở 4-bit NF4 (bitsandbytes), các lớp nn.Linear bị thay bằng Linear4bit. Code custom của DFlash draft (trust_remote_code) có thể đang check isinstance(module, nn.Linear) để biết tầng nào cần KV injection. Sau quantization, check này trả về False — injection có thể bị skip âm thầm (silent failure), không crash, không error, chỉ acceptance rate về 0.

synthetic_probe.py
import torch
from transformers import AutoTokenizer, AutoModelForCausalLM, BitsAndBytesConfig
from transformers import AutoModel

# ── 1. Load Target với 4-bit NF4 ──────────────────────────────
bnb_cfg = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.bfloat16,
    bnb_4bit_use_double_quant=True,
)

target = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen3-4B",
    quantization_config=bnb_cfg,
    device_map="cuda:0",
)
target.eval()
tok = AutoTokenizer.from_pretrained("Qwen/Qwen3-4B")

# ── 2. Forward pass ngắn, hook vào layer DFlash dự kiến dùng ─
probe_input = tok(
    "The factory produced 847 units. How many units were made?",
    return_tensors="pt"
).to("cuda:0")

hidden_states_captured = {}
def make_hook(name):
    def hook(module, inp, out):
        if isinstance(out, tuple):
            hidden_states_captured[name] = out[0].detach()
        else:
            hidden_states_captured[name] = out.detach()
    return hook

# Hook vào last hidden layer — DFlash thường dùng đây làm H_target
handle = target.model.layers[-1].register_forward_hook(make_hook("last_layer"))

with torch.no_grad():
    out = target(**probe_input, output_hidden_states=True)

handle.remove()

# ── 3. Kiểm tra H_target ──────────────────────────────────────
h = out.hidden_states[-1]
print(f"H_target shape  : {h.shape}")      # expect [1, seq_len, hidden_dim]
print(f"H_target dtype  : {h.dtype}")      # phải là bfloat16 hoặc float16, KHÔNG float32
print(f"H_target device : {h.device}")     # phải là cuda:0
print(f"H_target has NaN: {torch.isnan(h).any()}")   # phải False
print(f"H_target norm   : {h.norm().item():.4f}")    # phải > 0

# ── 4. Load Draft và chạy inference có KV injection ──────────
draft = AutoModel.from_pretrained(
    "z-lab/Qwen3-4B-DFlash-b16",
    trust_remote_code=True,
    torch_dtype=torch.bfloat16,
    device_map="cuda:0",
).eval()

# Nếu có DFlash generate API — gọi với dummy input và đo acceptance length
# Nếu acceptance_length > 0 → KV injection hoạt động qua Linear4bit
# Nếu acceptance_length == 0 → silent failure → cần patch trust_remote_code

vram_used = torch.cuda.memory_allocated("cuda:0") / 1e9
print(f"\nTotal VRAM used : {vram_used:.2f} GB")
print("Probe PASS nếu: dtype=bfloat16, no NaN, norm>0, acceptance_length>0")

Kết quả cần đạt: H_target dtype là bfloat16, không có NaN, norm > 0, và acceptance length > 0 khi chạy inference thật. Nếu acceptance length = 0 nhưng H_target intact → tìm đoạn isinstance(module, nn.Linear) trong custom code của DFlash và patch thành hasattr(module, 'weight') hoặc string-match tên class.

4. Chiến lược Nén Đầu vào

4.1 LLMLingua-2 với Answer-Aware Compression (MVP)

Giải pháp đúng cho bài toán "compressor cắt mất số toán học trong GSM8K" không phải là force_tokens (không thể liệt kê trước mọi số có thể có) và không phải manual chunking (phá vỡ global coherence). Giải pháp là: tách prompt thành segment bảo vệ và segment nén, kết hợp tham số question= để compressor biết phải bias importance score về phía gì.

compress_gsm8k.py — MVP pattern
from llmlingua import PromptCompressor

compressor = PromptCompressor(
    model_name="microsoft/llmlingua-2-bert-base-multilingual-cased-meetingbank",
    use_llmlingua2=True,
    device_map="cpu",           # LLMLingua-2 chạy CPU, không chiếm GPU VRAM
)

def compress_augmented_gsm8k(wiki_context: str, math_question: str, ratio: float) -> str:
    """
    wiki_context  : đoạn Wikipedia thêm vào (phần dài cần nén)
    math_question : câu hỏi GSM8K gốc (được bảo vệ, không qua compressor)
    ratio         : tỷ lệ GIỮ LẠI (0.33 = giữ 33%, tương đương R≈3)
    """
    compressed_wiki = compressor.compress_prompt(
        context=wiki_context,
        question=math_question,     # bias toward tokens relevant to answering
        rate=ratio,
        force_tokens=["\n", "?"],   # giữ cấu trúc xuống dòng và dấu hỏi
    )["compressed_prompt"]

    # Câu hỏi toán học KHÔNG đi qua compressor — nối vào sau
    return compressed_wiki.strip() + "\n\n" + math_question

# Ví dụ: ratio=0.33 → R_actual ≈ 3 (cần đo R_act thực tế sau khi chạy)
final_prompt = compress_augmented_gsm8k(wiki_chunk, gsm8k_q, ratio=0.33)
Tại sao pattern này đúng: (1) question= bias token classification về phía các số và đơn vị liên quan đến đáp án. (2) Câu hỏi toán học không bao giờ đi qua compressor nên không thể bị cắt. (3) LLMLingua-2's BERT base có hard cap 512 tokens — nó đã tự xử lý sliding window nội bộ cho input dài hơn, không cần manual chunking. Với scope 500–1.500 tokens của CC-DFlash MVP, đây là hoàn toàn đủ.

4.3 Gemma4-E2B Compressor — Phase 2, Nhánh Riêng

Gemma4-E2B là abstractive compressor (viết lại thay vì classify token). Không có nguồn nào trong tập DRR chứng minh nó tốt hơn LLMLingua-2 cho prompt compression. Rủi ro "mất số" khi abstractive compression là structural — không thể patch bằng force_tokens. Phải tách hoàn toàn sang nhánh nghiên cứu riêng sau khi LLMLingua-2 baseline chạy ổn. Không gộp chung vào MVP.

5. Phân tích Ttotal và Breakeven Condition

Ttotal = Tcompress + Tprefill(N/R) + Tdraft + Tverify + Tdecode
(1)
Tprefill(N/R) ≈ Tprefill(N)R²  — chỉ đúng cho prefill stage
(2)
Tprefill(N) · (1 − 1/R²) > Tcompress  ← Breakeven condition
(3)
Quan điểm đúng về Breakeven (từ phân tích phản biện): Nhiệm vụ của thực nghiệm là đo breakeven curve thực tế, không phải uốn nắn dữ liệu để breakeven nằm ở một mốc cụ thể. Nếu breakeven thực tế nằm ở N = 1.000 thay vì N = 500, đó vẫn là kết quả nghiên cứu có giá trị. Đặc biệt cần đo Tcompress thực tế với LLMLingua-2 trên CPU của môi trường cụ thể trước khi claim bất kỳ speedup nào.

5.1 Tác động đến Acceptance Rate α

‖ Htarget(Cgốc) − Htarget(Cnén) ‖ ≤ ε(R)
(4)

ε(R) tăng theo R. Giả thuyết R* ≈ 3 là điểm Pareto-optimal cần xác nhận thực nghiệm. Metric đo acceptance rate phải là Acceptance Length τ (theo chuẩn DFlash paper), không chỉ α (%).

5.2 Amortization trong Multi-turn / RAG

Tcompress,eff = TcompressQ → 0  khi Q → ∞
(5)

run_mvp.py — skeleton đầy đủ
import torch
from transformers import AutoTokenizer, AutoModelForCausalLM, AutoModel
from transformers import BitsAndBytesConfig
from llmlingua import PromptCompressor

# ── Target: 4-bit NF4 ──────────────────────────────────────────
bnb_cfg = BitsAndBytesConfig(
    load_in_4bit=True,
    bnb_4bit_quant_type="nf4",
    bnb_4bit_compute_dtype=torch.bfloat16,
    bnb_4bit_use_double_quant=True,
)
target = AutoModelForCausalLM.from_pretrained(
    "Qwen/Qwen3-4B",
    quantization_config=bnb_cfg,
    device_map="cuda:0",
).eval()

# ── Draft: BF16 (0.5B — không cần quantize) ────────────────────
draft = AutoModel.from_pretrained(
    "z-lab/Qwen3-4B-DFlash-b16",
    trust_remote_code=True,
    torch_dtype=torch.bfloat16,
    device_map="cuda:0",
).eval()

# ── Compressor: CPU ─────────────────────────────────────────────
compressor = PromptCompressor(
    model_name="microsoft/llmlingua-2-bert-base-multilingual-cased-meetingbank",
    use_llmlingua2=True,
    device_map="cpu",
)

# ── Tokenizer với Thinking Mode TẮT ────────────────────────────
tok = AutoTokenizer.from_pretrained("Qwen/Qwen3-4B")

def build_prompt(wiki_context, math_question, ratio):
    compressed_wiki = compressor.compress_prompt(
        context=wiki_context,
        question=math_question,
        rate=ratio,
        force_tokens=["\n", "?"],
    )["compressed_prompt"]
    combined = compressed_wiki.strip() + "\n\n" + math_question
    return tok.apply_chat_template(
        [{"role": "user", "content": combined}],
        tokenize=False,
        add_generation_prompt=True,
        enable_thinking=False,          # ← KHÔNG ĐƯỢC BỎ DÒNG NÀY
    )

# ── Đo VRAM thực tế ─────────────────────────────────────────────
print(f"VRAM sau load: {torch.cuda.memory_allocated('cuda:0')/1e9:.2f} GB")

8. Decision Log — v3.0 → v4.0

D1: VRAM Budget (lỗi lớn nhất của v3.0)

v3.0: Target ~2.5 GB FP16 + Draft ~1.5 GB FP16 = ~4 GB total
v4.0: Target 4.0B params → 8 GB FP16 → cần 4-bit NF4 → ~2.5 GB. Draft 0.5B params → 1 GB BF16 không cần quantize. Total ~3.5 GB + KV cache. Cần đo thực tế bằng Synthetic Probe.
Lý do: Qwen3-4B = 4.0B × 2 bytes = 8 GB FP16. Con số 2.5 GB v3.0 là không thể ở FP16.

D2: Thinking Mode

v3.0: Không đặc tả thinking/non-thinking cho benchmark
v4.0: enable_thinking=False khóa cứng trong mọi experiment. Draft Qwen3-4B-DFlash-b16 non-thinking only.

D3: Compressor Strategy cho GSM8K

v3.0: force_tokens=["\n", "?"] — không đủ để bảo vệ số toán học
v4.0: Segment separation + question= parameter (answer-aware compression). Câu hỏi toán học không đi qua compressor.

D4: Manual Chunking

v3.0: Không đề cập — rủi ro implicit nếu implement naively
v4.0: KHÔNG làm manual chunking. LLMLingua-2 tự xử lý BERT 512-token windowing nội bộ. Phase 2 dùng LongLLMLingua cho 4k+ tokens.

D5: Primary Metric DFlash

v3.0: α (%) — acceptance rate percentage
v4.0: τ (acceptance length) — metric hạng nhất của DFlash paper. α vẫn đo nhưng secondary.

D6: Gemma4-E2B trong Pipeline

v3.0: Gemma4-E2B fine-tuned là Compressor B trong cùng MVP pipeline
v4.0: Gemma4-E2B tách hoàn toàn sang Phase 2. Không để hypothesis phụ làm nhiễu MVP. LLMLingua-2 là compressor duy nhất trong MVP.

D7: Baseline Matrix

v3.0: AR / DFlash-R1 / CC-LLM / CC-Gemma
v4.0: Thêm "LLMLingua-AR" (compression không DFlash) để tách nguồn gốc speedup: đến từ compression hay từ speculative decoding?

D8: K-Flat TPU Claim

v3.0: "TPU v5p, chi phí verify K=1024 ≈ K=16 token" — dùng như fact nền tảng
v4.0: Claim chưa xác minh được nguồn chính thức. Xóa khỏi phần lập luận chính hoặc đánh dấu "unverified".

Phụ lục A: Ký hiệu Thống nhất

Ký hiệuĐịnh nghĩa
CoriginalContext đầu vào gốc (N tokens)
CcompressedContext đã nén (N/R tokens, text tự nhiên)
RCompression ratio = Noriginal / Ncompressed
R*Compression ratio Pareto-optimal — cần xác nhận thực nghiệm (giả thuyết R*≈3)
τAcceptance length (DFlash primary metric) — số token draft được accept trung bình mỗi verification step
αAcceptance rate (%) — secondary metric
HtargetHidden states Target Model từ Ccompressed — cần Synthetic Probe xác nhận qua 4-bit
TcompressThời gian nén — đo thực tế, không giả định
TprefillThời gian prefill Ccompressed qua Target Model (4-bit NF4)
ε(R)Sai số H_target(C_gốc) vs H_target(C_nén) — tăng theo R
